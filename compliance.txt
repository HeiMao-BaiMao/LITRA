1. DeepSeek 直接 (provider: "deepseek")
--------------------------------------------------------------------------------
[根拠] api-docs.deepseek.com/guides/thinking_mode: V3.2 以降 thinking モードは
ツール呼び出し対応。thinking 中は tool_choice required/特定関数指定が 400。
reasoning_effort の有効値は "high"/"max" のみ。thinking 中 temperature/top_p/
penalty は無視。マルチターンツールでは reasoning_content の返送必須
(@ai-sdk/deepseek 2.0.39 は返送対応済みを node_modules で確認)。

1-1 [A] provider-options.ts deepseek 分岐:
    - `if (toolsEnabled) → thinking disabled` の強制無効化を削除する。
      thinking の ON/OFF は deepseekThinkingEnabled のみで決める。
    - reasoningEffort は値が "high" または "max" のときだけ送る。
      それ以外("low"/"medium"/"xhigh" 等、旧設定の残骸)は送らない
      (サーバ既定 high に任せる)。
1-2 [A] service.ts isDeepSeekThinkingEnabled: toolsEnabled 引数を廃止。
    thinking ON ⇔ provider==="deepseek" && deepseekThinkingEnabled !== false。
    buildTemperatureOption / buildAdvancedOptions の呼び出しも追随
    (thinking ON なら tools の有無に関係なく sampling params を送らない)。
1-3 [A] tool_choice: 新ヘルパー resolveForcedToolChoice(settings) を service.ts
    に追加し export する。「ツール呼び出しを強制したい局面」で使う値を返す:
      - provider "opencode"                      → undefined (送らない)
      - deepseek で thinking ON                  → "auto"   (required は 400)
      - anthropic / github-copilot の claude-* で thinking が有効
        (fable-5 は常時有効扱い)                 → "auto"
        [根拠] Anthropic: thinking と tool_choice any/tool は非両立。
        AI SDK の "required" は {type:"any"} に変換される。
      - それ以外                                 → "required"
    適用箇所: service.ts の verifyToolCallNeed 後のリトライ(現 :730 の
    `s.provider === "opencode" ? undefined : "required"`)、main.ts :1749。
1-4 [A] verifyToolCallNeed: provider==="deepseek" のとき providerOptions
    { deepseek: { thinking: { type: "disabled" } } } を渡す
    (thinking の推論遅延で 15 秒タイムアウトに達し検証が常に失敗していた。
    非 thinking なら temperature 0.1 も有効になる)。タイムアウトは 15 秒のまま。
1-5 [B] capability.ts deepseek フォールバック / default-providers.json の
    deepseek-v4-flash / deepseek-v4-pro: supportedEfforts を ["high","max"] に。

2. Anthropic 直接 (provider: "anthropic") + GitHub Copilot の claude-*
--------------------------------------------------------------------------------
[根拠] claude-api 公式リファレンス(2026-06 版):
  - Opus 4.7 以降 / Fable 5: budget_tokens は 400。thinking は
    {type:"adaptive"} + output_config.effort (low/medium/high/xhigh/max)。
    thinking パラメータ省略時: Opus 4.7/4.8 は thinking OFF、Fable 5 は常時 ON。
    Fable 5 は {type:"disabled"} 送信も 400(=省略以外不可)。
  - Claude 4 以降: temperature と top_p の併送は 400(どちらか一方)。
  - thinking 有効時は tool_choice {type:"any"}/{type:"tool"} 非両立。
  - @ai-sdk/anthropic は opus-4-8/opus-4-7/fable-5 への temperature/topP/topK を
    自動除去する(node_modules で確認)が、sonnet-4-6/haiku-4-5 へは素通し。
    thinking の budget_tokens は全モデルで素通し(=SDK は守ってくれない)。

2-1 [B] default-providers.json claude-opus-4-8: reasoningCapability を
      { "kind": "anthropic-adaptive",
        "supportedEfforts": ["low","medium","high","xhigh","max"],
        "display": "summarized",
        "canDisable": true }
    に変更(budget 系フィールドは削除)。anthropicThinkingEnabled: false の
    既定は維持(= thinking パラメータ省略 = OFF)。
2-2 [B] capability.ts isThinkingAlwaysOn: `kind === "anthropic-adaptive" &&
    cap.canDisable !== true` に変更(Fable 5 = 常時 ON、Opus 4.8 = 切替可)。
2-3 [A] provider-options.ts anthropic 分岐(adaptive 側):
    - anthropicThinkingEnabled === false かつ canDisable な capability なら
      undefined を返す(thinking を一切送らない = Opus 4.8 では OFF)。
      Fable 5(canDisable でない adaptive)は従来どおり
      (effort 未設定なら undefined、設定時 {type:"adaptive"}+effort+display)。
    - thinking 有効時は effort 未設定でも {type:"adaptive"}(+display) を送る。
2-4 [A] service.ts buildAdvancedOptions: provider==="anthropic"、および
    provider==="github-copilot" かつ model が "claude-" 始まりのとき、
    topP / topK / frequencyPenalty / presencePenalty を送らない
    (temperature のみ送る。併送 400 の回避)。
2-5 [A] service.ts normalizeSettings: anthropicThinkingBudget が
    maxTokens 以上なら maxTokens - 1024 (最低 1024) に丸める
    [根拠: budget_tokens < max_tokens 必須、最小 1024]。
2-6 [A] service.ts: 非ストリーミングの背景ステップ(構想/査読/場面カード/
    話し方カード/選定/スパン修正)の maxOutputTokens を
    Math.min(settings.maxTokens, 32768) に制限する定数
    NONSTREAMING_MAX_OUTPUT_TOKENS = 32768 を導入
    [根拠: 非ストリーミングで 16K 超は HTTP タイムアウト域。DeepSeek の既定
     maxTokens 384000 がそのまま渡っていた]。
2-7 [B] settings-modal.ts: kind==="anthropic-adaptive" で canDisable な場合、
    既存の thinking ON/OFF コントロール(budget 用)を流用して ON/OFF を
    出せるようにする(最小実装でよい。無理に UI を新設しない。
    不可能なら現状維持で報告)。budget 入力欄は budget kind のみに表示。

3. OpenAI 直接 (provider: "openai") / Codex (provider: "codex")
--------------------------------------------------------------------------------
[根拠] developers.openai.com モデルリファレンス(2026-07):
  - GPT-5.1 以降 "minimal" は存在しない("none" が後継)。送ると 400。
  - GPT-5.6 (sol/terra/luna) は "max" が追加: none/low/medium/high/xhigh/max。
  - GPT-5.4 / 5.5: none/low/medium/high/xhigh。
  - reasoning summary は "auto" 推奨("detailed" はスナップショットにより
    拒否された実績あり)。
  - Codex backend (chatgpt.com/backend-api/codex) はパラメータ許可リスト制。
    store:false 必須。マルチターンツールは include:["reasoning.encrypted_content"]
    必須(無いと 2 ターン目に 400 "reasoning without its required following item")。

3-1 [A] provider-options.ts openai 分岐: reasoningSummary "detailed"→"auto"。
    openaiReasoningEffort === "minimal" は "none" として送る(旧設定救済)。
3-2 [A] provider-options.ts codex 分岐: 3-1 と同じ変換に加えて
      store: false,
      include: ["reasoning.encrypted_content"],
    を openai providerOptions に追加する。
3-3 [B] capability.ts openai/codex/copilot(gpt-5) フォールバック:
    ["none","low","medium","high","xhigh"]("minimal" を削除)。
3-4 [B] default-providers.json: gpt-5.6-sol/terra/luna(openai・codex・
    github-copilot 全セクション)の supportedEfforts を
    ["none","low","medium","high","xhigh","max"] に。gpt-5.5 / gpt-5.4 は
    ["none","low","medium","high","xhigh"](reasoningCapability が無い
    エントリには追加する)。gpt-5.4-mini は ["none"] のまま。
3-5 [C] codex-auth.ts createCodexFetch: sample/opencode 内の Codex プラグイン
    実装(sample/opencode/packages/ 以下で "originator" を grep して特定)を
    正確に写し、必要なヘッダ(originator、OpenAI-Beta、session_id 等、
    sample が送っているもの全て)を付与する。値も sample と同一にする。
    Authorization / ChatGPT-Account-Id の既存処理は維持。

5. OpenCode Go (provider: "opencode")
--------------------------------------------------------------------------------
[根拠] (a) DeepSeek V4 は reasoning_content を「専用フィールドのまま」返送必須。
<think> タグ折り込みは不可(公式明記)。OpenCode 本家 issue #24114 等で同型
400 多発。(b) sample/opencode/packages/opencode/src/session/llm/request.ts:187-195
のとおり、本家クライアントは opencode 系プロバイダーに
  x-opencode-project / x-opencode-session / x-opencode-request /
  x-opencode-client / User-Agent: opencode/<version>
を送る。これが無いと匿名扱いで厳格レート制限(earendil-works/pi#2824)。
(c) GLM-5/MiniMax は content が空文字の assistant+tool_calls メッセージを拒否。
(d) @ai-sdk/deepseek は modelId に "deepseek-v4" を含むとき V4 用の
reasoning_content 返送(未取得時 "")を行う(node_modules で確認)。

5-1 [A] provider.ts createModel opencode 分岐: OpenAI 互換経路のモデル
    (OPENCODE_GO_ANTHROPIC_MODELS に無いもの)は createOpenAI().chat ではなく
    createDeepSeek(common)(settings.model) で生成する。
    → reasoning_content のパース(表示)と返送が SDK ネイティブで行われる。
    deepseek-v4-flash/pro は V4 返送要件も満たす。GLM/MiMo も reasoning_content
    形式のため同じクライアントで動く。
5-2 [A] provider.ts: createOpenCodeReasoningTagStream / rewriteOpenCodeReasoningLine
    と、debugFetch 内の pipeThrough 適用(現 :730-739)を削除する
    (reasoning_content を content に潰す変換は 5-1 と両立しない。
    Anthropic 経路モデルには元々 reasoning_content が無く無害だった)。
5-3 [A] provider.ts debugFetch: isOpenCodeRequest のとき以下のヘッダを付与:
      x-opencode-client: "cli"
      x-opencode-session: アプリ起動毎に生成する固定 ID (例 "ses_" + ランダム)
      x-opencode-request: リクエスト毎に生成する ID (例 "msg_" + ランダム)
      x-opencode-project: インストール毎に安定な ID (例 "prj_" + ランダム。
                          localStorage 等に永続化できれば永続化、不可なら
                          モジュールレベル固定で可)
      User-Agent: "opencode/" + sample/opencode/packages/opencode/package.json の
                  version 値(実装時に確認して埋め込む)
    既存ヘッダを上書きしない形で set する。
5-4 [A] provider.ts debugFetch の opencode リクエスト正規化: リクエスト body の
    messages[] で role==="assistant" かつ tool_calls があり content === "" の
    要素は content: null に置換する(GLM-5.2 の空 content 拒否対策)。
    JSON パース不能時は無変換(既存の try/catch 方針に合わせる)。
